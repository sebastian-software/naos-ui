# Getting Started

Iktia v0.1 is a prerelease compiler toolchain for building native Custom
Elements from a React-like TSX subset. You write typed component functions; the
browser receives platform code.

## Install

```sh
pnpm add @iktia/core @iktia/runtime
pnpm add -D @iktia/compiler @iktia/vite @iktia/cli
```

`@iktia/compiler` installs one optional native package for the current platform.
There is no npm source-build fallback in v0.1.

## Configure TypeScript

```json
{
  "compilerOptions": {
    "jsx": "react-jsx",
    "jsxImportSource": "@iktia/core",
    "types": ["vite/client"]
  }
}
```

## Configure Vite

```ts
import { defineConfig } from "vite"
import { iktia } from "@iktia/vite"

export default defineConfig({
  plugins: [iktia()],
})
```

Vite owns modules, chunks, assets, cache invalidation, and CSS imports. Iktia
only transforms accepted `.wc.tsx` component modules.

## Author A Component

```tsx
import { computed, event, state, type ComponentOptions } from "@iktia/core"
import css from "./counter.css?inline"

export const options = {
  shadow: true,
  styles: [css],
} satisfies ComponentOptions

export function Counter({ label = "Count" } = {}) {
  const count = state(0)
  const text = computed(() => `${label}: ${count()}`)
  const change = event<number>("change", { bubbles: true })

  return (
    <button
      part="button"
      data-count={count()}
      onClick={() => {
        count.update((value) => value + 1)
        change.emit(count())
      }}
    >
      {text()}
    </button>
  )
}
```

The compiler infers the native Custom Element tag from the file and export. Host
pages consume the result with regular DOM APIs.

## CLI

```sh
iktia compile src/counter.wc.tsx -o dist/counter.js
iktia prerender src/counter.wc.tsx --props '{"label":"Static"}'
iktia info
```

Use the CLI for smoke tests, standalone transforms, and static HTML output.
Application builds should use the Vite plugin.

## Static HTML And DSD

Declarative Shadow DOM is the default HTML output for `shadow: true` components
inside the prerender path:

```sh
iktia prerender src/counter.wc.tsx --props '{"label":"Static"}' -o dist/counter.html
```

In Vite builds, prerender metadata is on by default so a static site or demo
build can discover compiled Iktia components:

```ts
iktia({
  prerender: {
    manifestFile: "iktia-manifest.json",
  },
})
```

The snippet above only customizes the manifest path. `iktia()` already uses
`iktia-manifest.json`. Use `prerender: false` only when a build should not emit
static HTML metadata.

There is no component-level DSD switch. If you prerender a `shadow: true`
component, Iktia emits `<template shadowrootmode="open">` host HTML and the
client module hydrates it on upgrade.

## Native Install Checks

If the compiler cannot load its native binding, run:

```sh
iktia info
```

The output shows the detected platform and native package. Published installs
should load the matching `@iktia/compiler-*` package. Repository development
uses a local binding built with:

```sh
pnpm -w build:native
```

## What Iktia Is Not

Iktia is not a React compatibility layer, a virtual DOM renderer, a bundler, or
a framework adapter. Vite owns modules, chunks, assets, and CSS loading. Iktia
only compiles accepted `.wc.tsx` modules into native Custom Elements.
