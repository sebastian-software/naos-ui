# Iktia v0.1 Milestone Plan

Status: Accepted for implementation planning
Date: 2026-06-16

This document is the execution plan for the first public Iktia prerelease. It
turns `docs/product-readiness-roadmap.md` into a decision-complete milestone
plan. The roadmap remains strategic context; this file is the implementation
source of truth for v0.1.

v0.1 targets installable public packages, clear docs, native binary
distribution, structured diagnostics, source maps, a minimal CLI, Ardo-rendered
docs, and linked static demos.

The core direction is React-like TSX authoring, native Custom Element output,
no React or Vue runtime, no virtual DOM, and no Iktia-owned module graph.

## Key Decisions

### Public Authoring API

v0.1 removes the early legacy authoring surface before any prerelease stability
promise is made.

Public API in v0.1:

* `state()`
* `computed()`
* `effect()`
* `event()`
* `on()`
* `host()`
* `Show`
* narrow typed `.map()` list authoring
* `ComponentOptions`
* Iktia JSX runtime types

Removed before v0.1:

* `component()`
* `prop.*()`
* `prop()`
* `signal()`
* `useHost()`
* public `<For>` docs if typed `.map()` fully covers the v0.1 list path

`state()` is the public local state primitive. Compiler internals may still use
signal or reactive naming where that is useful, but public docs, examples, and
types must use `state()`.

Event options are split by purpose:

* `event(name, customEventOptions)` controls `CustomEvent` dispatch defaults.
* `on(type, handler, listenerOptions)` controls `addEventListener()` listener
  options.

### Lists And Components

Public list authoring uses a narrow React-like `.map()` shape:

```tsx
items().map((item) => <Row key={item.id} label={item.label} />)
```

The compiler accepts only statically analyzable keyed map expressions. Dynamic
lists must provide a stable `key`. Accepted maps lower into keyed list IR; Iktia
does not add a virtual DOM runtime.

Iktia does not own a module graph. The compiler transforms module-local TSX.
Vite/Rolldown own graph traversal, chunking, assets, watch mode, and
invalidation. Iktia may read direct `.wc` imports for local JSX lowering, but it
must not become a bundler.

### Styling And Declarative Shadow DOM

CSS uses Vite `?inline` imports for v0.1:

```ts
import css from "./button.css?inline"

export const options = {
  shadow: true,
  styles: [css],
} satisfies ComponentOptions
```

The v0.1 CSS contract is intentionally flat:

* no Iktia CSS graph;
* no CSS Modules contract;
* no Sass or PostCSS contract;
* no custom Iktia CSS helper;
* no constructable stylesheet contract.

Component CSS is injected as `<style>` into Shadow DOM and Declarative Shadow
DOM output. Theming should use CSS custom properties, for example
`var(--iktia-accent, fallback)`.

Declarative Shadow DOM is public only through explicit prerender entry points:

* `renderDeclarativeShadowDom()` in `@iktia/compiler`
* `iktia prerender` in `@iktia/cli`
* optional Vite metadata needed for static demo builds

Normal client builds do not use a public `ComponentOptions.dsd` flag.
`data-iktia-*` markers are internal generated markup and are not
semver-protected. Development hydration mismatches throw structured diagnostics.
Production hydration mismatches remount imperatively.

### Packages And Distribution

Public v0.1 packages:

* `@iktia/core`
* `@iktia/runtime`
* `@iktia/compiler`
* `@iktia/vite`
* `@iktia/cli`

`@iktia/runtime` is a tiny public platform-helper runtime. It may contain event,
scheduling, and hydration helpers used by generated output. It must not contain
a virtual DOM, reconciler, component runtime, hook runtime, or framework
compatibility layer.

`@iktia/cli` publishes the `iktia` binary. v0.1 commands:

* `iktia compile`
* `iktia prerender`
* `iktia info`

Native package topology follows the Palamedes model. `@iktia/compiler` is the
platform-aware TypeScript wrapper and public compiler API. Platform-specific
optional packages carry compiled N-API artifacts:

* `@iktia/compiler-darwin-arm64`
* `@iktia/compiler-darwin-x64`
* `@iktia/compiler-linux-arm64-gnu`
* `@iktia/compiler-linux-arm64-musl`
* `@iktia/compiler-linux-x64-gnu`
* `@iktia/compiler-linux-x64-musl`
* `@iktia/compiler-win32-arm64-msvc`
* `@iktia/compiler-win32-x64-msvc`

Normal npm installs do not compile native code from source. Source builds are
contributor-only and remain documented for repository development. `iktia-core`
and `iktia-node` stay unpublished to crates.io for v0.1.

### Baselines

* Node.js: `>=22.0.0`
* Browser policy: modern evergreen browsers
* Core polyfills: none
* Browser test gate: Chromium, WebKit, and Firefox through Playwright

### M2: Public API Cleanup

Deliverables:

* Remove public exports for `component()`, `prop.*()`, `prop()`, `signal()`,
  and `useHost()` from `@iktia/core`.
* Update compiler analysis to reject removed authoring APIs with explicit
  diagnostics.
* Update examples and docs to use `state()` only.
* Update type tests to prove removed APIs are unavailable.
* Update `docs/compiler-limitations.md`, `docs/authoring.md`, README, and demos.

Acceptance criteria:

* No public example imports or calls removed APIs.
* `state()` is the only public writable local state primitive.
* Unsupported legacy syntax fails loudly instead of silently compiling.

### M3: React-Like List Authoring

Deliverables:

* Add support for narrow keyed `.map()` TSX list expressions.
* Lower accepted map expressions into keyed list IR.
* Require `key` for dynamic list elements.
* Reject unkeyed maps, arbitrary conditional/list expressions, unsupported
  block bodies, and non-JSX map returns with structured diagnostics.
* Remove public `<For>` docs if `.map()` fully covers v0.1 list authoring.

Acceptance criteria:

* `items().map((item) => <Row key={item.id} />)` compiles.
* Dynamic maps without `key` fail with a hint.
* Unsupported map shapes fail before code generation.
* Generated output updates keyed lists without introducing a virtual DOM.

## Test Plan

v0.1 is ready for public prerelease when these checks are in CI and green:

* Type tests verify public API exports and removed legacy APIs.
* Rust fixtures cover accepted `state()`, `computed()`, `effect()`, `host()`,
  `event()`, `on()`, keyed `.map()`, CSS styles, DSD, and rejected patterns.
* Compiler API tests cover `map?`, structured errors, source spans, and native
  typegen drift.
* Native loader tests cover every package name, Linux libc detection,
  unsupported platform errors, missing optional dependency errors, and workspace
  local binding.
* CLI tests cover `compile`, `prerender`, `info`, stdout/file output, exit
  codes, and structured error rendering.
* Playwright tests run public demos in Chromium, WebKit, and Firefox.
* Release dry-run verifies native package publish order before JavaScript
  package publish.

## Explicitly Deferred

* Forms, Form-Associated Custom Elements, and React-inspired form actions.
* Iktia-owned module graph or bundler replacement.
* CSS Modules, Sass, PostCSS, constructable stylesheet contract, or recursive
  CSS graph owned by Iktia.
* React, Vue, or Angular host demo matrix.
* crates.io publishing for Rust crates.
* npm install source-build fallback.
